### Introduction

This guide outlines the steps to integrate Nessus scanning logs with Wazuh. This integration relies on parsing the internal activity and audit logs generated by the Nessus scanning engine to track service status, user actions, and the lifecycle of vulnerability scans.

### Prerequisites

Before starting, ensure the following:

  * A functioning Nessus scanner generating activity logs.
  * A Wazuh Manager (or a designated Wazuh Agent) installed and configured to read the Nessus log file.

### Initial Nessus Configuration

To forward Nessus logs to Wazuh, you must configure a Wazuh agent (or the manager itself, if installed on the same machine) to monitor the local Nessus log file. This is typically done by adding a `<localfile>` block to your `ossec.conf` file to read the backend logs (e.g., `nessusd.messages` or the equivalent backend log file depending on your operating system).

Ensure the log collection is active so Wazuh can ingest the raw log lines and begin parsing fields like `nessus_user`, `nessus_action`, `nessus_scan_id`, and `nessus_duration`.

### Installing and Configuring the Integration

#### Copy the Custom Decoders

The custom decoders instruct Wazuh on how to parse the standard Nessus log format. Copy the decoders file to the Wazuh Manager's custom decoders directory:

```bash
cp integrations/nessus/custom_nessus_decoders.xml /var/ossec/etc/decoders/
```

Or navigate to **Server Management** --\> **Decoders** --\> **Add new decoders file** --\> paste the content, save the file and reload the cluster.

#### Copy the Custom Rules

The custom rules map the decoded fields to specific security and auditing alerts, such as alerting when a user starts, pauses, or stops a scan, or when a scan finishes. Copy the rules file to the Wazuh Manager's custom rules directory:

```bash
cp integrations/nessus/custom_nessus_rules.xml /var/ossec/etc/rules/
```

Or navigate to **Server Management** --\> **Rules** --\> **Add new rules file** --\> paste the content, save the file and reload the cluster.

#### Set File Permissions

Ensure that the Wazuh user has the appropriate permissions to read the newly added files:

```bash
chown wazuh:wazuh /var/ossec/etc/decoders/custom_nessus_decoders.xml
chown wazuh:wazuh /var/ossec/etc/rules/custom_nessus_rules.xml
chmod 660 /var/ossec/etc/decoders/custom_nessus_decoders.xml
chmod 660 /var/ossec/etc/rules/custom_nessus_rules.xml
```

#### Restart the Wazuh Manager (If using CLI)

If you copied the files directly via the command line, apply the new decoders and rules by restarting the Wazuh Manager service:

```bash
systemctl restart wazuh-manager
```

### Integration Testing

#### Test Using Wazuh-Logtest

You can verify that the decoders and rules are working correctly by using the `wazuh-logtest` tool with a log from your `sample_logs.txt`:

1.  Navigate to **Server Management** --\> **Ruleset Test** on the Dashboard.
2.  Paste a raw log from Nessus into the prompt (e.g., `[Tue Mar 10 15:30:48 2026 +0100][3812.10295244] User usuario requested STOP for scan 8401d1a5-f575-7163-e4f6-b5991febdb0a962aa9dedcb5ad3`).
3.  Verify that Phase 2 correctly identifies the `nessus_custom` decoder (and any child decoders like `nessus_custom_user_req`) and Phase 3 triggers the appropriate rule (e.g., Rule `110053` for a user stopping a scan, or `110052` for a started scan).

#### Verify in Wazuh Dashboard

1.  Open the Wazuh Dashboard.
2.  Navigate to the **Discover** tab.
3.  Add a filter: `Field` `rule.groups`, `Operator` `is`, `Value` `custom_nessus`.
4.  Alternatively, filter by `Field` `decoder.name`, `Operator` `is`, `Value` `nessus_custom`.
5.  Save the filter or view the dashboard to monitor ingested Nessus events in real-time.
